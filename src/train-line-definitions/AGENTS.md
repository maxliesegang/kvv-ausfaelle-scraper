# Train Line Definition Agents

Guidance in this file applies to `src/train-line-definitions/**`.
This is the most specific guidance for train-line definition source files.

## Scope

This area defines and loads train-number to line mappings used during parsing.

## Schema

Each `docs/<fahrplan-year>/train-line-definitions/<line>.json` file is a
`TrainLineDefinition` (see `types.ts`):

```jsonc
{
  "line": "S31",
  "trainNumbers": ["85101", "85103", "85696"], // flat list of all Zugnummern for the line
}
```

- `trainNumbers` is a flat, numerically-sorted list of every train number (Zugnummer)
  known for the line. It is seeded in full from GTFS — see
  `scripts/seed-train-lines-from-gtfs.ts` (`npm run seed-train-lines`) — and is not
  modified at runtime.
- A train number may legitimately appear under several lines: sibling lines (S5/S51/S52)
  share number blocks, and GTFS reuses one Zugnummer for distinct trips on connected
  lines (e.g. an S5 trip and an S6 trip at Pforzheim). Each line keeps the number.

Lookup (`src/train-lines.ts`): single-line and line-prefixed articles use the article's
own line. For a multi-line article a shared number is disambiguated by timing against the
`ambiguous-trips.json` sidecar (below); only when timing cannot anchor a run does it fall
back to **every mentioned line the number runs on** (the article's mentioned lines ∩ the
number's GTFS lines). A number that maps to none of the mentioned lines fails loudly with
`MultiLineMappingError` — there is no learning and no guessing.

## Ambiguous-trip sidecar

~10% of numbers run on more than one line, and the flat lists cannot say which line a
given cancellation belongs to. `docs/<year>/train-line-definitions/ambiguous-trips.json`
(schema and matcher in `ambiguous-trips.ts`) records, per shared number, every GTFS trip's
`{ line, dep, arr, dates }` signature. `resolveAmbiguousTrip` matches the article's date
and the trip's departure/arrival times (±2 min tolerance, midnight-aware) to report
exactly the line(s) of the one physical run: **one** line for a recycled number (separate
trains sharing a Zugnummer, e.g. 85055 as an S53 and a separate S6), **several** for a
through-running train that changes line mid-run (e.g. 85411 S7→S4). It is generated by the
same seed script, and is the reason through-running needs no override.

`overrides.ts` is a small lookup-time escape hatch (normally empty) for the rare number
GTFS maps to none of an article's mentioned lines — a feed gap or KVV-vs-GTFS conflict
that would otherwise be a hard parse error. It is not for through-running trains.

## Core Rules

- Mapping versions are tied to Fahrplan years.
- Runtime definitions are expected under `docs/<fahrplan-year>/train-line-definitions/`.
- Train numbers may change between Fahrplan years, so avoid cross-year assumptions.
- Re-seed a year with `npm run seed-train-lines -- <gtfs.zip> --year=<n>`; it overwrites
  that year's per-line lists **and** the `ambiguous-trips.json` sidecar with pure GTFS data
  (overrides are applied at lookup time, not baked into the files).

## Fahrplan Year Maintenance

When a new Fahrplan period is introduced:

1. Update year boundaries in `src/fahrplan.ts`.
2. Create `docs/<new-year>/train-line-definitions/`.
3. Optionally seed from previous year and then adjust.

If a date is outside known Fahrplan periods, the scraper should fail with a clear message prompting update of `src/fahrplan.ts`.
